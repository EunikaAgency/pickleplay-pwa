## 3. Anomalies

### A1 · CRITICAL — Create-game court bookings are invisible to the payer and orphan on delete
- **Where:** Create-game flow (`createBooking → checkout → createGame`); S2.3 / S2.10; reproduced again with B's game during cleanup.
- **Repro:** Create a game (pays and books the court). Open Games → Booking, Profile → My bookings, or `GET /api/v1/bookings`: the paid booking is absent everywhere — only the venue's hour grid shows it taken. Delete the game. Re-check availability.
- **Observed:** `GET /venues/:id/availability` still reports the hour taken after deletion — `2026-06-18` hour 9 `free:0` (A's deleted game) and `2026-06-13` hour 11 `free:0` (B's deleted game). No user-facing record, no cancel path, no refund messaging. The hour is dead inventory for the venue.
- **Expected:** The payer sees the booking under My bookings (it's their ₱800), and deleting the game cancels (or explicitly transfers) the booking.
- **Evidence:** [S2.3-booking-tab-reloaded](screenshots/e2e-2026-06-11/S2.3-booking-tab-reloaded.png) (UI shows only the old booking), [S2.3-jun18-availability](screenshots/e2e-2026-06-11/S2.3-jun18-availability.png) (hour blocked), [S2.10-jun18-after-game-delete](screenshots/e2e-2026-06-11/S2.10-jun18-after-game-delete.png) (still blocked post-delete), plus the availability JSON in section 1 of this run's logs.
- **Hypothesis:** the API stores game-flow bookings under a different owner/shape than `listBookings` queries (or filters game-linked bookings out), and `deleteGame` doesn't cascade to its `bookingId`.

### A2 · MAJOR — Expired token + optionalAuth = logged-in user's data silently vanishes
- **Where:** Games list (`GET /api/v1/games`, optionalAuth); hits My Games, home hero, Browse join-state. Seen repeatedly from ~15 min after login.
- **Repro:** Log in, host/join a game, idle past the access-token TTL, open Games → My Games.
- **Observed:** "You haven't created or joined any games yet" while hosting a published game. The expired token doesn't 401 on optionalAuth routes — the server answers 200 *as anonymous*, so the client's refresh-on-401 never fires and renders a false empty state. A reload (which refreshes via `/me`) restores everything. My verification calls hit the same trap twice.
- **Expected:** Stale token → refresh → correct data; never an empty "mine" list for an authenticated session.
- **Evidence:** [S2.10-state-check](screenshots/e2e-2026-06-11/S2.10-state-check.png) (empty My Games while hosting), [S2.10-my-games-back](screenshots/e2e-2026-06-11/S2.10-my-games-back.png) (same screen after token refresh).
- **Hypothesis:** optionalAuth middleware treats invalid/expired identical to absent. Fix server-side (401 on *present-but-invalid* token) or client-side (proactive refresh before expiry).

### A3 · MAJOR — PWA install banner overlays and blocks bottom CTAs
- **Where:** Court detail ("Book this court"), manage game ("Save changes"), game detail ("Join Game"), create club ("Continue") — any screen with a sticky bottom CTA.
- **Observed:** `.install-prompt` (fixed, z-50) sits on top of the CTA and intercepts taps — Playwright's click logs prove pointer interception on 4 different screens; a real thumb hits the banner area too. Dismissal is not persisted: the banner returns on every reload/new session.
- **Expected:** Banner stacks above the CTA (content inset), auto-hides on CTA screens, and stays dismissed.
- **Evidence:** [ANOMALY-install-prompt-blocks-cta](screenshots/e2e-2026-06-11/ANOMALY-install-prompt-blocks-cta.png).

### A4 · MAJOR — Past games are listed as joinable everywhere
- **Where:** Games → Browse, venue "Games here", home "Open games near you".
- **Observed:** On Jun 11, Browse shows sections "TUE JUN 9" and "WED JUN 10" with "3 left" and active Join; anonymous and authed `GET /games?status=published` both return them — the list has no date floor.
- **Expected:** Browse shows upcoming games only (or clearly separates "past").
- **Evidence:** [S2.4-browse](screenshots/e2e-2026-06-11/S2.4-browse.png), [S1.1-court-details](screenshots/e2e-2026-06-11/S1.1-court-details.png).

### A5 · MODERATE — Paid booking cancels on a single tap, no confirmation, no refund info
- **Where:** My bookings → Cancel (S1.6); same from the Booking tab.
- **Observed:** One tap flipped a ₱1,600 confirmed booking to "Cancelled" instantly — no confirm dialog, no undo, no refund messaging. Game **deletion** does have an inline confirm, so the pattern exists.
- **Evidence:** [S1.6-cancel-step1](screenshots/e2e-2026-06-11/S1.6-cancel-step1.png).

### A6 · MODERATE — Full games disappear from Browse for everyone
- **Where:** `GET /games` list (server-side) → Browse, guests and members alike.
- **Observed:** Once B filled A's game, it vanished from every Browse list (anonymous + authed verified); `GET /games/:id` still serves it. The filter sheet's "Has open spots" toggle implies full games should be visible by default. A full game becomes unfindable except via My Games or a direct link — you can't show a friend the lobby.
- **Evidence:** S5.1 guest Browse shows 3 games while 4 published exist; anonymous list JSON captured in-run.

### A7 · MINOR — Booking status labels conflict between surfaces

Evidence: [S1.3-games-booking-list](screenshots/e2e-2026-06-11/S1.3-games-booking-list.png) vs [S1.3-my-bookings](screenshots/e2e-2026-06-11/S1.3-my-bookings.png).

### A8 · MINOR — Back re-enters completed wizards / loses sub-tab context
- Back from My bookings re-opened the **finished booking wizard** at step 1; Back from club details re-opened the **finished create-club wizard**; Back from manage-game lands on the Games tab but resets to the Booking sub-tab. Completed flows should be popped off the nav stack.

### A9 · MINOR — Guest Games tab defaults to an auth-only view
- Cold-start guest → Games tab → "Couldn't load bookings / Authentication required" full-screen error (a raw API message) from an unauthenticated `GET /bookings`. Should default guests to Browse or show a sign-in prompt. Evidence: [S0.1-guest-games-tab](screenshots/e2e-2026-06-11/S0.1-guest-games-tab.png).

### A10 · MINOR — Silent kicks, deletions, and leaves
- B was kicked and his game deleted with **no notification** either time (the only producer is `game_full`); hosts likewise learn of leavers only by roster diff. For a locked-in "final, non-refundable" commitment model, lifecycle notifications matter.

### A11 · MINOR — Host can't delete a club in the app, but the error copy tells them to
- The host-leave rejection says "delete the club instead", and `DELETE /clubs/:id` works (used for cleanup) — but ClubDetails offers no delete control. Dead-end instruction.

### A12 · MINOR — Club feed/members fetch once per mount
- Switching About/Members/Feed doesn't refetch and there's no pull-to-refresh, so cross-account changes (new posts, likes) stay invisible until the screen remounts. Evidence: B's feed missed A's post until a full reload ([S4.4-bob-likes-alice](screenshots/e2e-2026-06-11/S4.4-bob-likes-alice.png) vs [S4.4-bob-feed-fresh](screenshots/e2e-2026-06-11/S4.4-bob-feed-fresh.png)).

### A13 · Accessibility & hygiene notes
- Closed bottom sheets stay in the accessibility tree as `dialog`s — including an auth sheet reading "You'll need an account to ." (empty intent) — screen readers can land in them.
- The checkout banner's icon ligature leaks as literal text "science" before "Test mode".
- The like button's label flips from "Like" to a bare count with no `aria-pressed`, so liked-state is color-only.
- Seeded venue images include literal `Unknown` and social-media page URLs fetched as images (ORB-blocked console noise on Nearby/home).
- Every ~15 min, the first API call after token expiry 401s before the refresh-retry succeeds — works, but logs a console error on most sessions (related to A2).

## 4. UX & product suggestions (ranked)

1. **Show game-court bookings in My bookings and cascade on delete** (fixes A1). As A, I paid ₱800 in the create flow and had no record of it anywhere afterwards; deleting the game silently kept the court blocked. Even if "the game *is* the receipt" is the intent, deletion must release or refund the hour.
2. **Refresh tokens proactively** (or make optionalAuth 401 on invalid-but-present tokens) (fixes A2). As A, watching my own hosted game disappear from My Games mid-session reads as data loss.
3. **Re-home the install banner** (fixes A3): inset it above sticky CTAs, hide it on flow screens, persist dismissal. It currently fights the user's primary action on at least four screens.
4. **Confirm + refund copy on paid-booking cancel** (fixes A5): match the delete-game inline confirm; state what happens to the money.
5. **Date-floor the games list** (fixes A4) — server-side, so the venue page and home rails inherit it.
6. **Keep full games in Browse with a "Full" badge** (fixes A6) and let "Has open spots" do its job; full lobbies are social proof, not noise.
7. **Add kick/delete/leave notifications** (fixes A10): the model locks players into commitments — tell them when the commitment changes under them.
8. **Split payment state from booking state** in labels (fixes A7): "Paid · Upcoming" beats "Complete" for a future reservation.
9. **Pop completed wizards off the back stack & preserve sub-tabs** (fixes A8).
10. **Guest Games tab → default to Browse** (fixes A9); never show guests a raw "Authentication required".
11. **Give club hosts the delete control the error message promises** (fixes A11).
12. **Refetch club feed on tab re-entry / add pull-to-refresh** (fixes A12).

Friction notes: the booking and create-game wizards are genuinely fast (3 steps, sensible defaults, live cost math — the end-time picker correctly forbidding ranges across booked blocks is a nice touch). The notification tap-through and the grace-period modal copy ("final and non-refundable") are clear. The DUPR explainer sheet on onboarding is good context where it's needed.
