# Launch Audit — 20 July 2026

Verification of the 16 July meeting action items against the codebase.
Scope: `app/`, `api/`. Method: code inspection, no runtime testing.

Status vocabulary: **DONE** (verified in code) · **PARTIAL** (works, gaps noted)
· **MISSING** (no implementation) · **DEFERRED** (decided out of launch scope).

---

## P0 — Launch blockers

### PB-01 · Open Play approval missing on organizer/venue sessions
**MISSING.** There are two Open Play systems. Player-hosted games have a correct
approval flow; organizer/venue-published `OpenPlaySession` has none.

- `api/src/features/content/content.controller.ts:1061-1063` — every tap creates
  `status:'registered'` immediately. Comment: *"No capacity gate and no waitlist."*
- `content.controller.ts:1094-1099` — `canAccessOpenPlayChat` returns true for any
  registration row, so lobby + chat are open to anyone who tapped join.
- `content.model.ts:67` — status enum is `['registered','waitlisted']`, no `pending`.
  (Compare `TournamentRegistration` at `:190`, which does have `pending`.)
- `manageOpenPlayRegistration` (`:1180`) has approve/decline branches, but nothing is
  ever created pending, so the approve path is unreachable.

This is the monetized path — organizer and venue sessions are where the platform earns.

**Decision (20 Jul):** build the full flow — pending state, host approve/reject,
server-side capacity enforcement, lobby gated to accepted players only.

### PB-02 · Player-game approval defaults to OFF
**PARTIAL.** `api/src/features/games/games.model.ts:60` —
`requiresApproval: { type: Boolean, default: false }`. Out of the box, joining
auto-confirms, contradicting the meeting requirement.

**Decision (20 Jul):** flip the default to ON.

### PB-03 · Double-booking race condition
**PARTIAL.** Check-then-insert with no transaction and no unique index.
Acknowledged in-code at `api/src/features/bookings/bookings.controller.ts:466-471`:
*"two near-simultaneous requests for the last open court could still both pass."*
All `Booking` indexes are non-unique (`bookings.model.ts:78-80`).

**Fix:** compound unique index on `{courtId, date, startTime, subUnitIndex}`
partial-filtered to non-cancelled. Low effort relative to impact.

### PB-04 · `pending_approval` bookings never expire
**PARTIAL.** `expireOverdueBookings` (`bookings.controller.ts:345-354`) only expires
`awaiting_payment`, and runs **lazily** — only when a player lists or fetches their own
bookings (`:369`, `:682`). No cron. `findSlotConflict` excludes only `cancelled`
(`:154`), so a silent owner blocks a court indefinitely.

### PB-05 · Booking rejection notifies nobody
**PARTIAL.** No `rejected` status exists; rejecting means setting `cancelled`.
`notifyUser` is called only in the `awaiting_payment` branch
(`api/src/features/venues/venues.controller.ts:2164`). The player is never told.

### PB-06 · Live-mode payment is a dead end
**PARTIAL.** Test mode simulates success against a hardcoded demo card. With
`paymentTestMode=false`:
- Bookings: payment stays `pending` forever, booking never confirms —
  `api/src/features/payments/payments.controller.ts:122-141`.
- Subscriptions: **role is granted anyway against an unpaid, provider-less payment** —
  `api/src/features/partner-subscriptions/partner-subscriptions.controller.ts:158-169`.
  A user gets coach/organizer access in production without paying.

Blocked on the client payment decision (GCash manual vs semi-automated vs gateway).

### PB-07 · Staff permissions unenforced server-side
**PARTIAL.** `owner.pricing.manage` and `owner.reports.view` have **zero
`hasPermission` call sites in `api/`** — UI-gated only.
- `POST /venues/:id/slot-overrides` (`venues.controller.ts:1466`) requires only
  `requireVenueManager` → any front-desk token can set slot pricing.
- `requireVenueOwner` (`:437-441`) resolves true for staff via `effectiveOwnerId`, so
  `PATCH /venues/:id` — which carries `priceFrom`, `weekendPrice`, `holidayPrice`,
  `memberDiscountPercent` (`:69,76-77,80`) — is reachable by staff.

Enforced correctly elsewhere: analytics (`:2234`), staff management (`:1206,1231`),
venue delete (`:448-455`). This is a targeted gap, not a systemic one.

## P1 — Required for launch

### PB-08 · Senior / PWD / discounted pricing
**MISSING — 0% built.** No hits for `senior`, `pwd`, `discountCategory`, `customerType`
anywhere in `app/src` or `api/src`. The only discount axis is venue-wide
`memberDiscountPercent` (`venues.model.ts:213`, applied `pricing.ts:173-179`).

Statutory 20% discount in PH — compliance exposure, not just a feature gap.
Current workaround is hand-typing an override amount
(`OwnerManualReservationScreen.tsx:126,203-205,406`) — untyped and unreported.

**Decision (20 Jul):** build the full category system — customer-category model,
per-venue rate configuration, and a category selector in **both** owner manual booking
and player self-serve.

### PB-09 · Venue duplicate detection at creation
**PARTIAL.** Claiming works end-to-end with admin review
(`venue-management.controller.ts:51-70`, `:80-96`; admin queue `AdminClaimsScreen.tsx`).
But `app/src/features/owner/OwnerNewVenueScreen.tsx:170-178` is a static banner —
typing a name searches nothing, and `createVenue` (`venues.controller.ts:605`) has no
duplicate-name guard. Owners can freely duplicate directory venues.

### PB-10 · Persistent player filters
**MISSING, and actively defeated.** `GamesScreenV2.tsx:146` uses plain component state;
`:448` **resets filters on every section switch**. No localStorage, no server-side
preference model (nothing in `api/src/features/settings` or `users`).

The filter model itself is rich and already covers the meeting's list —
`features/games/gameFilters.ts:24-45`. This is purely a persistence gap.

### PB-11 · Map clustering
**MISSING.** No cluster library installed (`leaflet` + `react-leaflet` only, no
`leaflet.markercluster`/`supercluster`). Every venue renders as an individual marker
across all four maps. "Cluster" appears only in comments.

Partially mitigated by zoom framing (`NearbyScreenV2.tsx:74-127`, `maxZoom:14`), which
helps off-screen pins but not co-located overlap — the original complaint.

Also: a full-screen map exists at `/map` (`FullMapScreen.tsx`, `App.tsx:829-830`) but is
**not linked from anywhere**.

### PB-12 · Coach verification dead end
**PARTIAL.** Profiles are created `isVerified:false, isListed:false`
(`coaches.controller.ts:289-290`), `isVerified` is not in the update schema (`:54`), and
no endpoint sets it — only seed scripts do. Discovery filters `{isListed:true}` (`:182`).

Net: a self-registered coach is permanently invisible, while
`CoachSubscribeScreen.tsx:36,43` advertises a "Verified Coach badge" that has no runtime
path. Paying users get nothing.

## P2 — Should fix before launch

### PB-13 · Design tokens not centralized
The v2 palette is re-declared **8 times**, once per screen scope —
`v2.css:13, 357, 1059, 2122, 3088, 4039, 4655, 5479`. The file's own comment at `:11-12`
admits it. Plus **203 hardcoded hex literals in `.tsx`**, concentrated in owner screens
(`OwnerPricingScreen` 21, `OwnerCalendarScreen` 16, `OwnerPartnersScreen` 15) — exactly
the surface Marvin's redesign must reach. A third, entirely stale palette lives in
`docs/DESIGN-TOKENS.md:5-40` (teal `#2FAFC3`, matches nothing).

Changing brand blue today is 8+ edits. Consolidate before the redesign lands, not after.

### PB-14 · Duplicate Book Court navigation
Four map/discovery screens: `nearby`, `map`, `search`, `owner-venues` — the last two
share a component (`App.tsx:757`). Two booking screens: `book-court`, `open-play-book`.
Six separate `book-court` call sites in `CourtDetailsScreen.tsx`. Home's "Book Court"
card and the Nearby tab land on the identical screen.

Dead code: `features/venues/NearbyScreen.tsx` (v1, 33k, unreferenced) and its
`NearbyFilterSheet.tsx`.

`docs/NAVIGATION.md` is stale — describes a 5-tab bar (actual is 6, `V2Chrome.tsx:92`)
and says deep linking is unwired (it is wired, `navigation.ts:119+`).

### PB-15 · Booking lifecycle gaps
- **No-show: MISSING.** Only a config knob `Venue.noShowFee` (`venues.model.ts:244`).
  No status, no endpoint, no charge logic.
- **Refund: PARTIAL.** Quote + 3-day policy exist (`bookings.controller.ts:704-729`);
  test mode auto-marks refunded; live mode returns the string
  *"Processing — 5–10 business days"* with no actual refund (`:770`).
- **`modifyBooking` hardcodes `priceDelta = 0`** (`:882`) — rescheduling a weekday slot
  onto a weekend surge slot is free. The `BookingModification.priceDelta` field is dead.
- **Reschedule conflict check is weaker than create** — passes `turnoverMinutes = 0` and
  `isSplittable = false` (`:856-866`), so it can land inside a turnover buffer that a
  fresh booking would have rejected.
- `Booking.status` has **no enum validation** (`bookings.model.ts:23`, plain String).
  Seed data writes `completed` (`seed-dummy-data.ts:430`), which no controller ever
  sets — orphan state.

### PB-16 · Notification coverage uneven
Infrastructure is real and dual-channel — Web Push + FCM (`shared/lib/push.ts:44-77`,
`shared/lib/firebase.ts`), persisted + SSE (`shared/lib/notify.ts:30-42`).
Silently no-ops if VAPID env vars are unset (`push.ts:15-19`).

Wired: `booking_pending_approval` (owner), `booking_approved` (player).
**Not wired:** booking confirmed, cancelled/rejected, payment received, payment window
expiring, refund processed.

### PB-17 · Manual booking writes a stray price override
`OwnerManualReservationScreen.tsx:268-273` writes a `price: 0, note:'Reserved'` slot
override as a side effect of every manual booking. Pollutes the pricing-override grid
and may zero-rate that slot for other bookers.

### PB-18 · Unreachable completed features
`OrganizerHubScreen.tsx:41-46, 61-67` — Tournaments and Venue Requests tiles commented
out ("HIDDEN: temporarily disabled for all roles"). `VenueRequestsScreen.tsx` (142 lines,
fully wired) and the entire `organizer/tournaments/` tree including the bracket generator
are complete code with no entry point. Decide: ship or delete.

### PB-19 · Dead file with fake stats
`features/profile/ProfileScreen.tsx:73-77` hardcodes `Games: '124'`, `Win rate: '66%'`.
Not routed by `App.tsx` (live path is `ProfileScreenV2`, which correctly shows real
metrics). Delete so it cannot be mistaken for live data.

## DEFERRED

### PB-20 · Court data import — **UNDONE, deferred (20 Jul)**
The importer works and fully supports courts (`import-real-data.ts:357-372`), but
`api/real-data/handoff/venue_courts.csv` is **67 bytes — header only, zero rows**.

Current state: **181 real venues, 0 real courts.** Venues carry only a scalar
`courtCount` (`:252`); there is no fallback synthesizing Court docs from it.

Any claim that "venue and court data is imported" is half-true. Held pending the
client's latest spreadsheet — deliberately **not** worked around during development.

### PB-21 · `courtId` optional server-side
`bookings.model.ts:6-7` — `venueId` is required, `courtId` is not. Court-level conflict
guards only run inside `if (body.courtId)` (`bookings.controller.ts:211, 454`).
The UI always sends a court (`BookCourtScreen.tsx:246-249` auto-selects `courts[0]`), so
the "court is the bookable resource" invariant is enforced by the client, not the model.

Related to PB-20 — tighten once real court data lands.
